TypeScript Code Metrics Explained

Cyclomatic Complexity
Counts the number of independent paths through a function — each if, else, for, while, case, catch, &&, and || adds 1. A score of 1-5 is simple, 6-10 is moderate, 11-20 is complex and should be refactored, above 20 is a maintainability risk. TypeScript functions with high cyclomatic complexity are harder to test and understand.

Type Coverage Score
Measures the ratio of your code that has explicit or inferred type information vs. uses any. A high any count defeats the purpose of TypeScript — each any is an escape hatch that bypasses type checking. Tools like typescript-coverage-report and type-coverage measure this across your entire repo. This analyzer counts explicit any usages as a proxy for coverage quality.

Maintainability Index
A composite score combining cyclomatic complexity, lines of code, and Halstead volume (a measure of code density). The FTA tool uses a similar scoring model. Higher scores mean more maintainable code. This analyzer grades from A (excellent) to F (needs immediate refactoring) using a weighted combination of complexity, file size, and type safety issues.

Function and Class Count
Large files with many functions are harder to navigate. As a rule of thumb: files should have a single responsibility, functions should do one thing, and classes should have a narrow interface. A TypeScript file with 20+ functions or 3+ classes is a signal to split into multiple modules.

FAQ — TypeScript Static Analysis
The main tools: ESLint + @typescript-eslint — most widely used, catches type-unsafe patterns and enforces best practices. tsc --strict — the compiler itself with strict null checks, no implicit any, and strict function types. FTA — Fast TypeScript Analyzer, a Rust CLI that measures complexity and maintainability scores across your codebase. SonarCloud — enterprise code quality with TypeScript-specific rules. knip — finds unused exports, dependencies, and files. This tool provides instant browser-based metrics with no install.
knip is the best dedicated tool for finding unused TypeScript exports, files, and dependencies across a project. ts-prune finds unused exports. ESLint's no-unused-vars and @typescript-eslint/no-unused-vars catch unused local variables at file level. TypeScript's own noUnusedLocals and noUnusedParameters compiler flags also detect unused declarations. This analyzer detects unused variables in the pasted code.
Three layers: (1) Compiler checks — enable strict: true in tsconfig.json for maximum type safety. (2) Linting — add ESLint with @typescript-eslint/recommended to catch patterns the compiler allows but are considered unsafe. (3) Complexity metrics — use FTA or this tool to measure cyclomatic complexity and flag files that need refactoring. Run all three in CI to enforce quality on every pull request.
FTA is an open-source TypeScript static analysis CLI written in Rust (using the swc parser). It analyzes TypeScript and JavaScript files for complexity and maintainability, outputting a score per file. It can analyze up to 1600 files per second. FTA is available via npm (npx fta-cli) and has a playground at ftaproject.dev. This browser tool provides similar per-snippet metrics without requiring installation.
Grade A (80-100): simple, well-typed, low complexity — easy to maintain. Grade B (65-79): minor issues, generally maintainable. Grade C (50-64): moderate complexity or type safety gaps — refactoring recommended. Grade D (35-49): high complexity or significant any usage — technical debt present. Grade F (below 35): difficult to maintain, high risk of bugs — prioritize refactoring. Most production TypeScript files should target Grade B or above.
